--------------------------------------------------------------------------------

SeaDrop is a portable, pocket-sized file transfer device powered by an ESP32-WROOM-32 microcontroller with an SPI TFT touchscreen and microSD slot. It creates its own WiFi access point, enabling an Android phone and a Windows laptop on completely separate networks to transfer files between each other without either device switching its primary internet connection or interrupting it in any way  --  at any point, including during active transfer.
SeaDrop is not a file transfer app. It is permanent wireless infrastructure you carry in your pocket. The line between both devices is always open. A transfer is just what happens when you use it.
1. System Overview
--------------------------------------------------------------------------------

1.1 Core Design Philosophy
SeaDrop's design is governed by one principle: the user should never think about the transfer mechanism. They have a file. They want it on the other device. Three taps from any context and it is done. SeaDrop handles everything else invisibly.
This is the gap AirDrop fills that LocalSend, KDE Connect, and every wireless transfer app fails to close. Those apps require both devices on the same network with the user actively managing a transfer UI. SeaDrop's interaction cost after one-time setup is: share from native context, tap destination, done.

1.2 Behavioral Requirements
 Neither device ever switches its primary WiFi or loses internet during a transfer  --  not during setup, not during transfer, not ever
 Neither user opens a transfer UI unless they choose to  --  share sheet and right-click context menu are the only required entry points
 SeaDrop is detected automatically when powered on  --  no manual reconnect after first setup
 The TCP connection is persistent  --  transfers start instantly because the line is already open
 After SeaDrop power cycles, both apps reconnect automatically within seconds  --  no user action
 After phone or laptop reboot, both apps reconnect on next boot  --  no user action
 One-time setup only: install app, complete first-launch wizard, done. No USB cable required for users

1.3 The Windows Single-Radio Problem  --  Solved
Standard Windows WiFi adapters  --  including the Realtek 8852BE-VT in the HP Victus (the reference hardware for this project)  --  report zero concurrent channels. This means connecting to a second WiFi AP drops the primary internet connection. This is a silicon-level constraint, not a driver issue.
The solution: invert the AP role. Instead of the laptop connecting to SeaDrop's AP, the Windows app programmatically creates a Mobile Hotspot with a fixed SSID and password known to the ESP32. The ESP32 connects to the laptop's hotspot in STA mode. The laptop stays on its home WiFi for internet. Mobile Hotspot and infrastructure WiFi coexist on the same single radio via the NDIS 'Infrastructure Station + Software AP' configuration  --  the same mechanism Windows uses for everyday Mobile Hotspot sharing. This works on the Realtek 8852BE and is confirmed behavior on this chip.
The ESP32 then bridges Android (on its own SoftAP) and Windows (via STA connection to the laptop hotspot) through the SeaDrop relay. Both platforms keep their internet. Neither touches a WiFi setting.

1.5 Why Neither Device Loses Internet
Android: WifiNetworkSuggestion adds SeaDrop's SoftAP as a secondary OS-managed network. Android detects the AP has no internet via captive portal probe  --  but the ESP32 NCSI spoof responds correctly, making Android believe the network has internet. bindProcessToNetwork() routes only the SeaDrop app TCP socket through the SeaDrop AP. The phone's original mobile data or WiFi is completely untouched for all other traffic.
Windows: The Windows app creates a Mobile Hotspot with a known SSID/password via NetworkOperatorTetheringManager API. The laptop stays connected to the home router on its physical WiFi radio. Mobile Hotspot runs concurrently via the NDIS Infrastructure Station + Software AP configuration  --  one radio, two concurrent network roles. The ESP32 connects to the hotspot in STA mode. No routing table manipulation. No WiFi Direct dependency. No concurrent channel requirement.

2.4 RSSI-Based Proximity Trust Zones
SeaDrop uses WiFi RSSI to determine proximity trust tier per connected device. The ESP32 reads RSSI directly from the SoftAP station table for Android, and from the STA connection stats for Windows. RSSI is averaged over 5 samples taken every 5 seconds. Tier change requires 3 consecutive readings in the new tier before committing.

3.2 Critical Hardware Decisions
8-bit parallel display is abandoned. It requires 13 GPIO pins including three strapping pins (GPIO 2, 12, 15), causes guaranteed boot failures, and offers no real speed advantage over hardware SPI with DMA on ESP32. SPI ILI9341 at 40MHz replaces it.
SPI microSD via SD.h is abandoned. Real-world throughput: 0.2 - 0.5 MB/s. A 100MB file would take 3 - 8 minutes. SDMMC 4-bit replaces it: 2.5 - 10 MB/s sustained write. The SPI display pivot frees exactly the GPIO pins required for SDMMC 4-bit.

3.4 Mandatory eFuse Burn  --  Day 1 Step 0
GPIO 12 is SDMMC D2. The SD specification requires a 10k pull-up on all data lines. But GPIO 12 HIGH at boot tells the ESP32 to set flash voltage to 1.8V  --  the WROOM-32 uses 3.3V flash, causing an immediate boot loop. A pull-down resistor fighting the SD pull-up causes CRC errors at speed.
The permanent fix: burn the eFuses to lock VDD_SDIO to 3.3V, making the ESP32 ignore GPIO 12 at boot forever. This is an irreversible one-time operation that must be done before any firmware is flashed:
   pip install esptool
   espefuse.py --port COM_PORT set_flash_voltage 3.3V
   # Confirm when prompted  --  this cannot be undone
After the eFuse burn, GPIO 12 can safely use the standard 10k pull-up required by the SD specification. SDMMC operates correctly at full speed without boot conflicts.

4.2 WiFi APSTA Mode
The ESP32 runs in WIFI_AP_STA mode simultaneously:
AP interface: Hosts SeaDrop SoftAP for Android client. SSID: SeaDrop-XXXXXX. WPA2. max_connection=3.
STA interface: Connects to Windows Mobile Hotspot. SSID and password stored in NVS from registration. Auto-reconnects on boot.
   WiFi.mode(WIFI_AP_STA);
   WiFi.softAP(SEADROP_SSID, SEADROP_PASS, channel, false, 3);
   WiFi.begin(HOTSPOT_SSID, HOTSPOT_PASS); // Windows hotspot
Channel matching: the Windows app reads the current home WiFi channel and passes it to the ESP32 via the first STATUS exchange. If the hotspot channel differs, the ESP32 restarts its SoftAP on the matching channel to minimize radio switching overhead on the Realtek adapter.

4.3 NCSI Spoofing  --  The Internet Trick
When any device connects to the ESP32 AP or when ESP32 connects to the Windows hotspot, the OS sends connectivity probes to verify internet access. Without a response, the OS marks the network as 'No internet' and may deprioritize or drop it. The ESP32 intercepts these probes and responds correctly, making both OS platforms believe the SeaDrop network has internet.
Implementation uses DNSServer to redirect all DNS queries to the ESP32's own IP, and ESPAsyncWebServer to serve the correct responses:
   // DNS: redirect everything to 192.168.4.1
   dnsServer.start(53, "*", IPAddress(192,168,4,1));
   
   // Windows NCSI probes
   // www.msftconnecttest.com/connecttest.txt
   server.on("/connecttest.txt", HTTP_GET, [](AsyncWebServerRequest *req){
       req->send(200, "text/plain", "Microsoft Connect Test");
   });
   // www.msftconnecttest.com/redirect
   server.on("/redirect", HTTP_GET, [](AsyncWebServerRequest *req){
       req->redirect("http://192.168.4.1");
   });
   
   // Android NCSI probes
   // connectivitycheck.gstatic.com/generate_204
   // clients3.google.com/generate_204
   server.on("/generate_204", HTTP_GET, [](AsyncWebServerRequest *req){
       req->send(204);
   });
   
   // Apple captive portal (future iOS support)
   server.on("/hotspot-detect.html", HTTP_GET, [](AsyncWebServerRequest *req){
       req->send(200, "text/html",
           "<HTML><HEAD><TITLE>Success</TITLE></HEAD><BODY>Success</BODY></HTML>");
   });

4.4 Samsung Galaxy A15 OneUI 8  --  Confirmed Compatible
The Samsung Galaxy A15 LTE running OneUI 8 (Android 16) uses standard Android NCSI behavior: HTTP GET to connectivitycheck.gstatic.com/generate_204, expects HTTP 204 response. The old Samsung S7 hardcoded DNS workaround (setting AP IP to 8.8.8.8) documented in pre-2019 repos is not required on OneUI 6 and later. The standard generate_204 --> 204 response from the ESP32 is sufficient.
NCSI on modern OneUI caches positive results per network (by BSSID). Once the probe succeeds on first connection, Android considers the network internet-capable for the session. The spoof must be running before the phone connects  --  the ESP32 starts DNSServer and ESPAsyncWebServer before activating SoftAP.

4.5 NCSI Cache Warning
Windows NCSI caches negative results per network BSSID. If Windows detects no internet on the SeaDrop hotspot before the ESP32 NCSI spoof is running, it will not retest that network for an extended period. Mitigation: the ESP32 starts the DNS server and HTTP responder in setup() before WiFi.softAP() is called. The spoof is live before any client can connect.

4.6 Streaming Relay vs SD Buffer
Mode 1: Streaming Relay: Both peers connected. Bytes flow directly from sender TCP socket to receiver TCP socket via in-memory ring buffer. SD not touched. Throughput approaches WiFi radio ceiling.
Mode 2: SD Buffer: Receiver offline. File written to SDMMC. Pushed to receiver on reconnect via NOTIFY/PULL. SD wear only in this mode.

SeaDrop Transfer Protocol (STP) v1.3
--------------------------------------------------------------------------------

Minimal persistent TCP protocol over port 4242. All commands are newline-terminated ASCII. Binary payloads follow immediately with byte length declared in the command. Connection is permanent  --  commands flow over the same open socket indefinitely.

5.3 Power Cycle Reconnect
SeaDrop powers off: both clients detect dead socket via PING failure or SO_KEEPALIVE timeout. Reconnect loop: 3 second backoff, retry indefinitely. SeaDrop powers on: SoftAP up in ~2 seconds. Android reconnects via WifiNetworkSuggestion. ESP32 STA reconnects to Windows hotspot. Windows TCP client reconnects. Both HELLO handshakes complete. Line live. End to end: under 10 seconds.

8. Setup Verification Gate
--------------------------------------------------------------------------------

Both platform wizards include a verification step before declaring setup complete. This catches NCSI spoof failures, hotspot compatibility issues, and driver problems on the user's specific hardware before they become invisible runtime failures.

8.1 Android Verification
 App connects to SeaDrop AP during registration
 App checks Android's network capabilities for the SeaDrop network  --  verifies NET_CAPABILITY_INTERNET is reported
 If NCSI spoof worked: capability is present, wizard proceeds
 If not: wizard shows 'Network check failed  --  SeaDrop may not stay connected on this device' with diagnostic steps

8.2 Windows Verification
 After ESP32 connects to Windows hotspot, app checks primary WiFi internet by making an HTTP request to a known external address
 If internet still live: pass. Wizard shows 'Internet confirmed intact.'
 If internet dropped: fail. Wizard shows exact failure mode and stops setup
 App also reads current WiFi channel and sends CHANNEL command to ESP32 for SoftAP channel matching
9. Security Model
--------------------------------------------------------------------------------

Auth tokens16-byte random tokens in NVS. Required on every HELLO. Unknown tokens dropped immediately.SoftAP passwordWPA2 shown on TFT during Registration Mode only. Prevents random AP joins.Hotspot passwordGenerated randomly during Windows registration. Stored in NVS and Windows Credential Manager. Not user-visible after setup.Registration Mode gateREG commands only in Registration Mode. 5-minute timeout. Partial registrations cleared on timeout.Physical proximityBLE ~10m range. Transfers are proximity-gated by radio physics.RSSI trust tiersWiFi RSSI gates auto-confirm behavior. UX convenience layer only  --  not a cryptographic boundary.Transfer consentReceiver accepts every incoming file. Far-tier requires explicit TFT tap. Close-tier auto-confirms after 3s with cancel visible.No cloudNo server, no relay, no internet routing. Files never leave the local SeaDrop network.Token storageAndroid: EncryptedSharedPreferences AES-256. Windows: Credential Manager DPAPI.NCSI spoof scopeNCSI spoofing only serves connectivity probe URLs. The ESP32 does not relay or proxy any other internet traffic. The spoof is a UX mechanism, not a network gateway.TransportPlaintext TCP behind WPA2. Acceptable for personal proximity transfers. TLS available on ESP32-WROVER (8MB PSRAM) for enterprise use.10. Sequential handled by outbound queue on app side.Android API floorWifiNetworkSuggestion requires API 29 (Android 10).WifiNetworkSuggestionManual disconnect via system WiFi picker blacklists suggestion for up to 24 hours  --  not just that session.
