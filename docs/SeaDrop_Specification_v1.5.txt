SeaDrop is a portable, pocket-sized file transfer device powered by an ESP32-WROOM-32. It creates its own WiFi access point, enabling an Android phone and a Windows laptop on completely separate networks to transfer files in both directions without either device losing internet at any point.
System requirements: Windows 11 24H2 (build 26100) or later. Android 16 (API 36) or later.

1. System Overview

1.1 Core Behavioral Requirements
Neither device loses internet at any point â€” not during setup, not during transfer, not ever
Bilateral file transfer: Android to Windows and Windows to Android, identical UX in both directions
Both apps work when SeaDrop is powered off â€” they drop the SeaDrop connection and resume primary internet silently
Both apps reconnect to SeaDrop automatically when it powers back on â€” no user action
Firmware works with or without TFT display attached â€” headless mode for flashing and testing
Firmware works with or without SD card â€” streaming relay still functions, buffer mode disabled gracefully

One-time setup only via first-launch wizard â€” no USB cable for users

3. SeaDrop Disconnect Behavior

When SeaDrop powers off or goes out of range, both apps must drop the SeaDrop connection cleanly and resume normal operation. The user should not notice anything except the tray icon going grey on Windows and the notification updating on Android.

3.1 Windows Disconnect Flow
ESP32 powers off or goes out of range
TCP keepalive or PING timeout detected in TcpListener â€” socket throws IOException
SeaDropService catches exception, marks state = DISCONNECTED
HotspotManager.StopTetheringAsync() â€” hotspot stops
Primary WiFi adapter resumes as sole active network interface automatically
Windows internet resumes â€” no user action, no route table cleanup needed
Tray icon goes grey. Tooltip: 'SeaDrop not found'
BleWatcher continues scanning for UUID 0xFEAD
When SeaDrop powers on: BLE detected â†’ HotspotManager.StartTetheringAsync() â†’ ESP32 connects â†’ TCP session re-established â†’ tray icon green

3.2 Android Disconnect Flow
ESP32 powers off or out of range
TCP keepalive or PING timeout â€” socket throws IOException
SeaDropService catches exception, marks state = DISCONNECTED
connectivityManager.bindProcessToNetwork(null) â€” releases SeaDrop network binding
Android OS drops SeaDrop AP from active connections automatically
Primary mobile data or home WiFi resumes as default route â€” no user action
Persistent notification updates: 'SeaDrop not found â€” searching...'
BleScanner continues scanning for UUID 0xFEAD
When SeaDrop powers on: BLE detected â†’ WifiNetworkSuggestion reconnects â†’ bindProcessToNetwork() â†’ TCP reconnects â†’ notification: 'SeaDrop connected'

3.3 User-Initiated Disconnect â€” Android
If the user manually disconnects from SeaDrop AP in system WiFi settings, Android blacklists the WifiNetworkSuggestion for up to 24 hours. The app detects this via NetworkCallback (LOSING state on the SeaDrop network) and shows a persistent notification: 'SeaDrop disconnected â€” tap to reconnect.' Tapping opens the system WiFi settings to the SeaDrop AP entry. Once the user connects once manually, the blacklist clears and auto-connect resumes.

3.4 Headless SeaDrop â€” No Display, No SD
If TFT is not attached, the firmware skips all TFT_eSPI init and all display_task rendering. The TCP server, BLE, NCSI, and relay all operate identically. The BLE payload includes a headless_mode flag so apps can optionally show 'SeaDrop running in headless mode' in their UI.
If SD card is not attached or fails to mount, the firmware logs the failure over Serial, sets sd_available = false, and disables buffer mode. Streaming relay continues to work normally. If a transfer is initiated while receiver is offline and SD is unavailable, ESP32 sends REJECT SD_UNAVAILABLE. The sending app shows: 'Receiver offline and SeaDrop has no storage. Try again when both devices are connected.'
// In sd_buffer.cpp

    Serial.begin(115200);

    // 1. NVS â€” load all credentials first
    nvs_store.begin();

    // 3. Optional: TFT display

    // 5. SoftAP

    // 7. BLE â€” includes headless and sd_available flags

    // 8. TCP server
    tcp_server.begin(4242);

    // 9. FreeRTOS tasks
    xTaskCreatePinnedToCore(tcp_server_task, "tcp", 8192, NULL, 5, NULL, 0);
    xTaskCreatePinnedToCore(ble_adv_task,    "ble", 4096, NULL, 3, NULL, 0);

7.2 wiFiControl and Location Permission â€” The Crash Fix
wiFiControl is a standard DeviceCapability, not a restricted custom capability. It works with a self-signed certificate installed to the machine's TrustedPeople store. The crash that occurs 3 seconds after launch is caused by calling CreateFromConnectionProfile before WiFiAdapter.RequestAccessAsync() has been called and granted. The OS requires explicit location consent before granting wiFiControl at runtime.
The fix is a strict startup gate:
// SeaDropService.cs â€” startup order is non-negotiable
public async Task StartAsync() {
    // Gate 1: location permission
    var access = await WiFiAdapter.RequestAccessAsync();
    if (access != WiFiAccessStatus.Allowed) {
        ShowNotification("SeaDrop needs location permission",
            "Open Settings > Privacy > Location and allow SeaDrop");
        return; // do not proceed
    }

    // Gate 2: start hotspot
    var started = await _hotspotManager.StartAsync();
    if (!started) {
        ShowNotification("Hotspot could not start",
            "Check that WiFi is enabled and try again");
        return;
    }

    // Gate 3: BLE watcher
    _bleWatcher.Start();

    // Gate 4: TCP listener â€” waits for ESP32 to connect
    await _tcpListener.StartAsync();
}

// SeaDropService.kt
override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
    startForeground(NOTIF_ID, buildNotification("Starting..."))
    scope.launch {
        // Gate 1: permissions
        if (!hasLocationPermission()) { requestPermissionsViaActivity(); return@launch }
        if (!hasNearbyWifiPermission()) { requestPermissionsViaActivity(); return@launch }
        if (!isBatteryExempt()) { requestBatteryExemptionViaActivity(); return@launch }

        // Gate 2: register WiFi suggestion
        wifiManager.registerSuggestion()

        // Gate 3: acquire locks
        wifiLock.acquire()
        wakeLock.acquire()

        // Gate 4: BLE scan
        bleScanner.start()

        // Gate 5: maintain TCP connection loop
        maintainConnection()
    }
    return START_STICKY
}

8.3 Maintain Connection Loop
private suspend fun maintainConnection() {
    while (true) {
        try {
            waitForSeaDropNetwork() // suspends until WifiSuggestion connects
            val network = connectivityManager.getNetworkForRequest(seadropRequest)
            connectivityManager.bindProcessToNetwork(network)
            socket = Socket().apply {
                keepAlive = true
                soTimeout = 15000
                connect(InetSocketAddress("192.168.4.1", 4242), 5000)
            }
            sendHello()
            checkOutboundQueue()
            updateNotification("SeaDrop connected")
            readLoop(socket!!) // blocks until dead
        } catch (e: IOException) {
            // SeaDrop went offline
            socket?.close(); socket = null
            connectivityManager.bindProcessToNetwork(null) // release binding
            updateNotification("SeaDrop not found â€” searching...")
            delay(3000)
        }
    }
}

10. Security Model

Auth tokens
16-byte random tokens in NVS. Every HELLO validated. Unknown tokens dropped.
SoftAP password
WPA2, shown on TFT during Registration Mode only.
Hotspot password
Generated at Windows registration. Stored in NVS and Credential Manager.
Registration gate
REG commands in Registration Mode only. 5-minute timeout.
RSSI trust
UX convenience layer only. Not a security boundary.
Transfer consent
Receiver accepts every file via notification or TFT.
No cloud
No server, no relay, no internet routing. Files never leave local subnet.
NCSI scope
NCSI spoof serves probe URLs only. No internet proxying.
Transport
Plaintext TCP behind WPA2. TLS available on ESP32-WROVER with 8MB PSRAM.
Version gate
REJECT VERSION_MISMATCH blocks transfers on incompatible firmware/app pairs.

12. Known Constraints

wiFiControl crash
Caused by missing WiFiAdapter.RequestAccessAsync() before CreateFromConnectionProfile. Fixed in v1.5 startup gate. Not a certificate restriction.
WifiNetworkSuggestion
Manual disconnect via system WiFi picker: 24-hour blacklist. App shows notification with manual reconnect prompt. Reconnecting once clears the block.
Realtek 8852BE-VT
Zero concurrent channels. Solved by laptop hosting hotspot, ESP32 connecting as STA. Co-existence confirmed in netsh wirelesscapabilities output.
Headless firmware
All display and touch tasks skipped. All network and transfer functionality intact.
No SD card
BUFFER mode disabled. REJECT SD_UNAVAILABLE sent. Streaming relay unaffected.
OEM battery killers
Samsung OneUI, Xiaomi MIUI, Huawei, OnePlus. Battery exemption required plus Adaptive Battery disabled for SeaDrop.
Android API floor
API 36. WifiNetworkSuggestion, NEARBY_WIFI_DEVICES, FOREGROUND_SERVICE_TYPE_DATA_SYNC all stable.
eFuse burn
Irreversible. Required for SDMMC D2 pull-up safety. Skip if SD card not used.
TLS
Not on WROOM-32. Available on WROVER 8MB PSRAM or ESP32-S3.
NCSI cache
Windows caches negative NCSI per BSSID. ESP32 starts DNS + HTTP before SoftAP to prevent cached negatives.
Version mismatch
REJECT VERSION_MISMATCH blocks transfer. App shows GitHub releases link. User reflashes firmware or updates app.
